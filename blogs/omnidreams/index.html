1背景与动机：为什么需要“生成式”世界模型

自动驾驶（AV）正进入新阶段：具备“推理”能力的视觉-语言-动作（VLA）模型可以一边思考一边规划轨迹。但核心难题始终是—— 如何在安全攸关的长尾场景中测试这些策略，再将其部署到实车。这需要一个既可交互又可扩展的仿真器。

1.1 开环 vs 闭环：到底差在哪？

这是理解整篇论文的基础。区别只有一句话：策略的动作会不会反过来影响它之后观测到的画面。

🔁 开环 Open-Loop（回放录像）
  • 向策略输入一段预先录制的真实驾驶日志，观测序列固定不变。
  • 策略每步输出动作 / 轨迹，与司机当时的真实动作比较（L2、ADE / FDE）。
  • 策略输出不反馈到环境：无论它如何决策，下一帧永远是录像里的下一帧。
  • ✅ 简单、可复现、成本低；❌ 无法暴露误差累积——一旦偏离便永远观测不到“偏离之后的世界”。
♻️ 闭环 Closed-Loop（动作真实驱动）
  • 动作真实驱动仿真：动作 → 更新世界状态 → 渲染新状态下的下一帧 → 反馈给策略 → 再出动作……
  • 策略会观测到自身造成的后果：压线后，下一帧即是压线视角。
  • 误差会累积并反馈，能暴露真正的失效（碰撞、驶离路面 off-road、轨迹蛇行 swerving）。
  • 指标是“结果型”：碰撞率、压线率、是否完成路线；❌ 代价是必须有一个能对任意动作渲染出正确观测的仿真器。
💡 一个具体例子
录像中专家在某弯道轻微向左转向。开环：即便策略选择直行，下一帧仍是“专家过弯后”的画面，单步误差很小、看似无异常。闭环：策略直行 → 仿真器将车辆置于直行后的位置 → 渲染出“驶向路肩”的全新画面 → 策略面对真实后果 → 大概率驶离路面。同一策略，只有闭环才能将这一错误放大并捕获。这正是 开环差距 open-loop gap / 协变量偏移（covariate shift）。
🔗 为什么这正是 OmniDreams 的切入点
闭环要求仿真器能渲染策略实际行驶的那条新轨迹上的画面——它可能大幅偏离原始录像。重建式（NuRec）一旦偏离采集轨迹画质便急剧下降（论文 Fig.14：偏离 4m+ 时 FVD 升至 207），而 OmniDreams 凭借生成先验在新轨迹上仍保持保真（~125）。同样的“开环→闭环”思想也出现在训练侧：teacher forcing（输入真值历史，类比开环）会导致曝光偏差，而 Self Forcing 在训练时即让模型闭环自我 rollout（见第 5 节）——这是同一协变量偏移原理的两次体现。

1.2 重建式 vs 生成式仿真

在“传感器渲染器”这一角色上，存在两条技术路线，OmniDreams 选择了后者：

🟠 重建式仿真（NeRF / 3DGS / NuRec）
  • 从真实采集数据重建场景，照片级逼真、贴近现实。
  • 支持在已重建场景内做 “what-if” 测试。
  • 根本局限：被原始采集数据牢牢锚定。一旦偏离采集走廊、需引入新内容、或切换到差异较大的条件，便难以泛化——新视角下出现模糊、鬼影、内容缺失。
🟢 生成式世界模型（OmniDreams）
  • 在海量视觉数据上学到世界如何演化的先验。
  • 能合成极具挑战性的现象：暴雨、暴雪、大风，以及车顶上的床垫等异常物体。
  • 能生成复杂的智能体行为：行人、随本车靠近而骤变的关节运动与制动。
  • 代价：相比重建式所需算力大得多。

四个关键设计

  • 自回归扩散视频生成式仿真器：与开源策略 Alpamayo 1、编排器 AlpaSim 组成闭环。
  • 历史 KV cache 复用：生成新帧时会注意到此前生成所缓存的 Key / Value，从而保证长序列 rollout 的一致性。
  • 条件化于仿真状态与驾驶动作：照片级生成被锚定在过去帧、当前仿真状态与当前动作上。
  • 实时交互渲染：2B 单摄像头版本在单卡 GB300 上 720p（704×1280）可达 68 FPS；2B 四摄像头版本在 16 卡 GB300 上可达 105 FPS。

2.2 三路条件输入（对应 Fig.3）

每生成下一步的传感器帧，OmniDreams 同时接收三类信号（第一步还额外接收一张 RGB 首帧）：

CONDITION 1
📝 文本提示
描述天气、光照、时段等环境属性，不改变几何。经 Cosmos 文本编码器，通过交叉注意力（cross-attention）注入。
CONDITION 2
🗺️ 抽象世界场景图
即 abstract state：渲染出的车道线、路杆等高精地图 + 动态物体的 3D 边界框（cuboid），编码了未来本车轨迹与他车行为。
CONDITION 3
🧠 记忆缓存
存储此前生成 token 的流式 KV cache，提供时序上下文，保证长时一致。
💡 直觉
文本控制“外观”（样貌 / 天气），抽象场景图控制“在何处、向何方”（几何 / 轨迹 / 他车），KV cache 维持“时序连贯”。 三者解耦，因此可以只改天气而不改道路，或只改轨迹而不改背景——这正是后文“可控反事实编辑”的基础。

3数据与构建

3.1 两套数据集

训练一个闭环世界模型对数据管线有特殊要求：需在结构化场景表征 + 本车轨迹 + 文本环境描述的条件下，合成照片级、时序一致的传感器观测。论文使用了两套真实路测数据：

  • RDS（Real Driving Scene）：3M 段 20s 片段，7 路同步摄像头，用于中训练（mid-training，介于预训练与后训练之间），同时包含在 Cosmos-Predict 2.5-AV 预训练集中。
  • RDS-HQ-1M：1.14M 段（10s / 20s 混合），高质量 + 精确的世界场景标注，用于微调与后训练。

3.2 条件信号是怎么抽取出来的

  • 世界场景图（抽象状态）：高精地图（车道线、路沿、路杆、停止线、人行横道、路面标记、红绿灯、交通标志，均来自预建的城市级地图并严格匹配到行驶路线）+ 对动态物体（车辆、行人、弱势道路使用者 VRU）做 3D 检测与跟踪（在 10 FPS 上检测、插值到 30 FPS）。再按每个相机视角渲染成像素对齐的线条 / 面 / 2D 投影立方体框；立方体框按朝向上色以提供深度线索。
  • 文本提示：用视觉-语言模型（VLM，Qwen2.5-VL-7B）对 10s 窗口、7 路摄像头分别生成描述文本（caption），每段生成短（~40 词）/ 中（~80 词）/ 长（~200 词）三种长度，按 0.1 / 0.2 / 0.7 概率采样，使模型对提示长度具备鲁棒性。

3.3 质量过滤与 SIL-Wheel 策展平台

多阶段过滤：剔除传感器 / 标注不可靠、预测分歧的片段；用 VLM 检测色差等视觉伪影；按本车轨迹 + 视觉特征去重（对高速直行这类重复片段做下采样）。 此外用 NVIDIA SIL 的视频检索与策展平台 SIL-Wheel 在片段（clip）粒度上完成三件事：① 构造后训练切片（上采样罕见天气 / 施工区 / 弱势道路使用者等）；② 构造留出评测切片（支持长尾、分场景报告）；③ 数据巡检（校验自动标注质量）。

4模型架构

4.1 模型有多大？参数量与规格一览

论文将参数量给为近似值（“~2B” / “~10B”），并未公布主 DiT 的精确层数、隐藏维度（hidden dim）、注意力头数（Fig.4 仅标注泛指的 “× N blocks”）。因此“2B”是论文报告口径的数量级，并非精确到亿位的数字。

4.2 因果自回归扩散与两个变体

🔑 为什么用“因果自回归扩散”而非常规视频扩散？
闭环要求即时响应动作变化。常规离线视频模型（Sora / Wan 等）采用双向、整段去噪，无法在线实时调整。OmniDreams 改用 因果扩散：每次预测仅依赖过去观测与当前条件；时序一致性由流式 KV cache 保证——复用已计算的注意力 K / V， 无需重算整段序列，从而在长时域上高效且稳定地 rollout。

两个变体：

  • OmniDreams-SV（单视角）：仅生成单个前向摄像头，每步 8 帧（= 2 个 latent 帧）。
  • OmniDreams-MV（多视角）：联合生成 4 路同步视角（前、左交叉、右交叉、前长焦），每步 16 帧（= 4 个 latent 帧）。

4.3 轻量控制分支（不是 ControlNet）

OmniDreams 不另设一个网络处理控制输入，而是把结构化仿真状态用一个小型 MLP 编码成紧凑的“控制 token”，与视觉 token 的隐变量表征对齐后拼接（concatenate），再送入 transformer。优点：计算开销极小；通过将控制信号与视觉内容清晰分离，稳定了自回归生成；并保持了实时所需的吞吐。

4.4 一致的多视角生成：注意力的二维分解

朴素做法是让所有视角、所有时间的 token 做全自注意力，复杂度高达 O(N²T²)（N = 视角数，T = 时长），实时不可行。OmniDreams 将注意力分解为两部分：

⏱️ 时序注意力（Temporal）

在每个视角内部独立进行。token 通过因果 KV cache 注意到本视角的过去帧，捕捉运动动态、维持时序一致。

👁️ 跨视角注意力（Cross-View）

在每个时间步跨视角进行。不同视角的 token 相互注意，使共享的几何、物体位置、运动在各摄像头间保持一致。

分解后总复杂度降至 O(NT²) + O(N²)，效率显著提升，支持更高分辨率、更长上下文、更多视角（最多 7 路环视）。 在结构上（Fig.4），相比单视角的 Cosmos-Predict 2.5 骨干，MV 的每个 Cross Block 额外加入：(i) 逐视角的视角嵌入，与时间嵌入相加，作为对 shift / scale / gate 的 AdaLN 调制；(ii) 在文本交叉注意力之后插入一层跨视角注意力。

5训练流水线：多阶段“改造” Cosmos

从双向的 Cosmos-Predict 2.5 出发，逐级将其改造成实时、可控、长时一致的因果世界模型：

STEP 1
AV 中训练
在 RDS 上中训练，提升自动驾驶领域能力。
STEP 2（仅 MV）
多视角适配
加入跨视角注意力层，学习跨摄像头对应与一致性。视角嵌入与跨视角输出投影零初始化以保证稳定收敛。
STEP 3
Diffusion Forcing
因果掩码训练，将双向模型转为自回归因果模型。
STEP 4
世界场景图控制
后训练阶段加入零初始化的控制分支，先训 93 帧再扩展到 189 帧以学习长时一致性。
STEP 5
Self-Forcing 蒸馏
实现少步生成，并用 DMD 消除自回归误差累积。
STEP 6
渐进式长上下文教师
用更长上下文的双向模型作为教师，继续蒸馏以抑制“漂移伪影”。

① 流匹配（Rectified Flow）目标

基础训练目标为流匹配：L = E[‖u_θ(x_t,t) − v_t‖²]，其中 x_t=(1−t)x+tε、v_t=ε−x，可附加条件 c（文本、首帧、控制信号）。

② Diffusion Forcing（因果化的关键）

沿用 Chen et al. 2024：对每个 token 独立采样一个扩散时刻 t_i（取自对数正态分布），并施加因果掩码，使 x^i 只能注意到 ≤ i 的帧。这样一次前向即可并行计算不同噪声级，天然支持自回归。还可推广到块自回归（k 帧一组），实现上采用 Flex-Attention。目标：L_DF = E[‖u_θ(x_t^{1:T}, t) − v_t‖²]。

③ Self Forcing + DMD 蒸馏（治“曝光偏差”）

⚠️ 曝光偏差（exposure bias）
标准 teacher forcing 在训练时每帧都看到干净的真值历史，但推理时只能看到自身生成的（含瑕疵的）输出 → 分布失配 → 长序列 rollout 的误差逐步累积放大。

Self Forcing（Huang et al. 2025）在训练时即让模型自我 rollout，每帧条件在自身此前生成的输出上（K=2 步扩散，时间步调度 [1000, 450]）。为保证可训练：梯度仅回传到随机抽样的单个去噪步，并从前帧的 KV cache 嵌入切断梯度；采用滚动 KV cache（保留最近 L 帧、驱逐最旧帧）将推理复杂度从 O(TL²) 降到 O(TL)，支持任意长度序列。

蒸馏目标采用 DMD（分布匹配蒸馏 Distribution Matching Distillation）：最小化自我 rollout 视频分布 p_θ 与真实分布 p_data 之间的反向 KL 散度：L_DMD = E[½‖x̂ − (f_ψ(x̂_t,t) − f_φ(x̂_t,t))‖²]，其中 f_φ 是冻结的真实 score、f_ψ 是学习的伪 score。score 之差在全序列视频级将生成器推向真实数据流形，无需逐像素的配对监督。

④ 渐进式长上下文教师（治“漂移伪影”）

当滚动 KV cache 在长生成中超出训练上下文窗口时，会出现累积的漂移伪影（shifting artifacts）。对策：先训练一个上下文窗口更长的双向模型（其不受滚动缓存问题影响），再将其作为教师，继续蒸馏先前由短上下文教师蒸馏过的模型。蒸馏阶段使用了 58k 段高质量、挑战性强（密集车流、复杂光照、复杂道路几何）的子集。

6推理优化（全部免训练）

6.1 五项推理优化

💡 关键
本节所有优化均为 免训练（training-free）——不改动任何扩散模型权重，仅优化推理系统。结果：MV 四视角 16 帧块在 16 卡 GB300 上 151ms（105 FPS / 摄像头）；SV 单视角 8 帧块在单卡 GB300 上 118ms（68 FPS）。
  • 局部时序注意力：固定窗口——SV 用 6 个 latent 帧（24 RGB 帧），MV 用 8 个 latent 帧（32 RGB 帧），在显存与速度间权衡。
  • 流式静态形状 KV cache：预分配固定大小，整段 rollout 张量形状不变；缓存更新置于独立线程，使主线程免受缓存更新延迟影响。
  • 编译 + CUDA Graphs：由于 KV cache 形状固定，DiT 前向运行在单一静态形状上，用 torch.compile 捕获为 CUDA Graph，首块惰性捕获、后续复用。

  • 上提步不变算子：将 RoPE 频率、patchify / unpatchify 等仅依赖块形状、与去噪步无关的计算，提到去噪循环之外，每块只算一次。

6.2 多 GPU 推理：三轴上下文并行

DiT 暴露三条天然的并行轴：视角轴 V、块内时间轴 T、单帧注意力内的空间轴 HW。采用层次化上下文并行，16 卡配置为 V=4, T=4, HW=1，分片顺序 V → T → HW。由于自注意力按摄像头独立、且是 DiT 的主导开销，沿 V 分片从不拆分单次自注意力调用——4 路 V 在 4 卡内近线性加速；其余工作优先用 T 分片（保持每时间步的跨视角注意力块完整），只有超过 16 卡时才对 HW 分片。上下文并行注意力采用自研的 ring-attention，将 KV 分片传输与本地注意力计算重叠。

6.3 FlashDreams：将这套推理栈开源

上述方案（流式静态 KV cache、局部窗口注意力、CUDA Graph 捕获）不限于本模型架构，已打包为开源流式推理栈 FlashDreams，并在 Wan2.1 系骨干上验证：Self Forcing 文生视频在单卡 GB200 上提速最高 1.95×；Lingbot-World（基于 Wan2.1-14B 的自回归相机控制世界模型）在 4×H100 上提速最高 2.49×。此外还附带端到端服务栈（gRPC + webRTC）。

7闭环集成与 AlpaSim

7.1 视频模型当渲染器的三大特性与挑战

闭环使用时，视频模型被打包为一个有状态的“视频模型服务器”，通过 gRPC 与仿真客户端通信（Fig.6 展示两次连续往返）：每次往返中，客户端发来一段轨迹 + 文本提示 +（仅首块）初始帧；服务器渲染世界场景条件、运行因果 DiT 的两步去噪 + 流式 KV cache、解码为 RGB、回传 JPEG 编码帧。服务器在整段 rollout 中保持状态（KV cache、CUDA Graph、渲染器状态均跨块复用），后续请求只需携带新轨迹与提示。

将视频模型用作渲染器，会带来三个工程挑战：

  • 分布式推理：高帧率下采用多进程（通常每 GPU 一进程）跨进程通信，无法像普通 Python 依赖那样内嵌进仿真层。
  • 自回归 → 内部有状态：不能以任意顺序渲染帧，仿真器必须跟踪该状态、避免失效。
  • 分块生成：当前视频模型按“块（chunk）”（对应 VAE 时间压缩）生成，意味着要把时间量化成区间，每个区间开始时智能体行为必须固定且已知。

7.2 AlpaSim 与网络化集成

AlpaSim 是开源、面向研究的 AV 仿真器，采用微服务架构、gRPC、Docker 部署，异步 RPC 天然带来流水线并行。OmniDreams 在其中替换掉 NuRec 充当相机帧渲染器。 做法是在视频模型部署的 rank 0 上运行一个 gRPC 服务器接收渲染请求，经 NCCL 事件转发给其余 rank；一块渲染完成后在 rank 0 聚合、序列化并回传 AlpaSim——这样无需将 OmniDreams 的依赖 / 代码 / rank 调度逻辑与 AlpaSim 其余组件强行统一。

7.3 Session 状态与分块生成的两种策略

将 KV cache 状态抽象成 session：每次 rollout 开启新 session（含种子首帧 + 世界地图），服务器分配 session ID 并预分配 KV cache，客户端在后续所有请求中携带该 ID。 由于一个视频块内的所有帧都须预先给定本车 / 他车轨迹（块中途不能更改，否则整块失效），论文比较了两种策略，AlpaSim 采用 pre-fetch：策略与交通模型在块边界一次性生成多步轨迹而非单步，按视频帧时刻插值后再请求视频模型——这样能严格保持事件时序顺序，避免隐蔽缺陷（对照的 post-fetch 会打乱事件时序）。

8把 OmniDreams 当成驾驶策略（WAM）

💡 一个反直觉的问题
OmniDreams 在生成式预训练中学到的内部表征，本身对“驾驶”是否有用？能否把同一个骨干微调成一个有竞争力的驾驶策略？答案：能，而且很强。

将 OmniDreams-SV（2B，120° 前广角）微调成端到端轨迹预测器，称为 世界-动作模型（World-Action Model，WAM）。从世界场景图微调之前的因果自回归检查点初始化，预测未来 6.4s、10Hz 的本车轨迹（64 个航点）。

架构改动（DiT 骨干保持不变，每帧增加两项）

  • DINOv2 特征注入：来自 dinov2_vitb14 编码器与 30° 前长焦摄像头的 patch 特征，经可学习的线性投影加到带噪视频隐变量 token 上，作为额外条件。
  • 历史 token：将过去 1.6s 的本车运动经小型 MLP 嵌入成一个 token，加入每帧 token 集、沿时间轴交错。注意力掩码保持视频-视频的因果性，对历史 token 施加单向耦合。该 token 的输出 h 送入 12 层 U-Net 形 MLP，参数化未来轨迹隐变量（τ ∈ ℝ64×3）的流匹配速度场。

推理时只运行一次 DiT，4 个流匹配步全部限制在轻量轨迹 MLP 内（无需视频去噪），因此极快。

9把 OmniDreams 当成“重建修复器”

第三个用途：用 OmniDreams 来改善已有的重建式仿真器。NeRF / 3DGS 重建对新视角外推很敏感——当相机位姿大幅偏离原始采集轨迹时，会出现模糊、内容缺失、鬼影、漂浮几何等视角相关伪影，降低闭环真实度，并给下游策略引入分布偏移。

做法：用“重建器产出的退化渲染 + 对应的干净目标图”构造配对数据，后训练一个蒸馏过的 OmniDreams 检查点。关键技巧——去噪过程不从随机高斯噪声开始，而是从退化渲染本身开始。模型由此学到一条流，将伪影污染的重建输出映射回干净图像流形，同时保持场景布局、相机视角与驾驶相关结构（Fig.7）。推理时它作为一个自回归修正模块叠加在预重建场景之上，依靠因果历史与 KV cache 维持时序一致——让重建式仿真器既保留其几何 grounding，又能享受生成式世界模型的视觉先验。

10.3 长尾覆盖与反事实变化

  • 可控场景编辑：通过修改文本提示（天气 / 光照 / 时段）、抽象世界场景图（车道几何 + 边界框）与首帧种子，单个训练好的检查点即可定向生成行人、雪天、夜晚 + 交通锥等变体（Fig.10），且未被编辑的属性（道路几何、远景）保持视觉稳定。
  • 分布外（OOD）物体建模：直接在首帧插入恐龙、长颈鹿等罕见物体，但它们在世界场景图中没有对应的 cuboid，导致信号冲突。对策——采用随机动态 cuboid dropout 后训练（训练时随机移除动态智能体的 cuboid，保留停放车辆等静态 cuboid），迫使模型从视觉历史 / 首帧 / 周边上下文推断物体的存在与运动。于是推理时插入的 OOD 物体能在生成视频中自然地传播运动（Fig.11）。

10.4 闭环评测：OmniDreams 是真实世界的“忠实代理”吗？

核心实验：在同一 AlpaSim 栈中仅替换传感器仿真器（NVIDIA NuRec vs OmniDreams），编排器、交通 / 物理服务、逐场景初始状态全部固定，于是 rollout 指标的差异就隔离出了传感器仿真器对策略行为的影响。使用 Physical AI AV NuRec 数据集的 501 场景子集，20s rollout、每 533ms 重规划，评测四个策略类（OmniDreams WAM、Alpamayo 1.5 的 4 / 2 / 1 摄像头变体）。

✅ 关键结论 1：策略排名得以保持（Fig.13）
在 NuRec 下策略的相对排名（WAM > Alpamayo 4cam > 2cam > 1cam），切换到 OmniDreams 下完全保持。 由于 NuRec 紧贴原始录制、是真实世界的强代理，这说明 OmniDreams 本身即为闭环策略评测的忠实代理—— 开发者在 OmniDreams 中比较候选策略，会得出与对照真实日志数据相同的结论。
✅ 关键结论 2：轨迹偏离越大，OmniDreams 优势越明显（Fig.14）
按“距原始录制轨迹的距离”分箱计算 FVD：随着新轨迹偏离真值，NuRec 的视觉质量迅速劣化（4m+ 时 FVD 升至 207）， 而 OmniDreams 几乎保持恒定（约 125），二者出现 82.2 的 FVD 差距。这正是重建式渲染器最吃力之处：新视角、光流变化、遮挡边界移动都需要外推到采集范围之外； OmniDreams 能利用学到的视频先验 + 世界场景条件，合成超出录制相机路径的合理观测。对动态物体的渲染也更自然（Fig.15）。

权衡：NuRec 只需标定首帧 + 抽象世界场景图（AV 数据管线即可产出），并暴露重建式无法原生提供的可控性 / 事故覆盖手段；但生成式仿真器原生所需算力远多于重建式——这是质量与算力之间的取舍。

11总结：贡献 · 局限 · 意义

11.1 主要贡献

  1. 首个实时闭环生成式 AV 世界模型：从 Cosmos 中训练 / 后训练，结合自回归 + 流式 KV cache + 少步蒸馏，将每块时延压入 100+ FPS，并真正接入开源策略（Alpamayo 1）与仿真器（AlpaSim）。
  2. 三路条件 + 轻量控制分支 + 注意力分解：实现可控、跨视角一致、最多 7 路环视的多视角实时生成。
  3. WAM 结果：同骨干微调成策略，2B 参数在闭环中超越 10B VLA——是首批证明“世界-动作模型在驾驶上可优于 VLA”的工作之一。
  4. 忠实代理验证：闭环中策略排名在 NuRec 与 OmniDreams 间保持一致，且轨迹偏离越远生成式越稳定。
  5. 开源系统 FlashDreams：通用的流式自回归视频 / 世界模型推理栈。

11.2 局限与权衡

  • 算力开销大：生成式仿真器原生比重建式昂贵得多（这也是需要大量推理优化的原因）。
  • 仍受分块语义约束：当前每块内的智能体行为需提前固定；逐帧生成是未来方向。
  • 长序列 rollout 仍有漂移：靠渐进长上下文教师抑制，但 Δ 仍不为零。
🎯 为什么重要
它标志着 AV 仿真从“纯重建 / 回放”走向“生成式世界模型”：能造出标准仿真器难以重现的长尾边缘场景， 为严格评测乃至训练下一代 VLA 策略提供可扩展工具，在“仿真测试”与“安全实车部署”之间架起桥梁。 更深一层——同一个能“生成世界画面”的模型也能“输出驾驶动作”，预示着世界模型与策略统一的可能。

📖术语速查表

闭环仿真 Closed-Loop
策略主动驾驶、动作改变环境、再接收新观测的循环；区别于回放录像的开环。
世界模型 World Model
学习世界状态如何随动作演化、并能据此生成未来观测的模型。
VLA / WAM
VLA = 视觉-语言-动作模型；WAM = 世界-动作模型，将视频输入映射到动作、无显式语言模态。
自回归扩散 / 因果掩码
每帧仅依赖过去帧与当前条件进行去噪生成，支持在线实时响应。
KV cache / 流式静态 KV cache
缓存注意力 Key / Value 以摊销跨步计算；后者形状固定，支持 CUDA Graph 与滚动驱逐。
Diffusion Forcing
每个 token 独立采样噪声级的训练范式，使一个模型既能做下一帧预测又能做整段去噪。
Self Forcing
训练时自我 rollout、每帧条件在自身上一帧的生成上，弥合训练-推理差距（治曝光偏差）。
DMD 分布匹配蒸馏
用真 / 伪 score 之差，在全序列视频级将生成分布推向真实分布，无需逐像素的配对监督。
注意力 sink token
KV cache 中保留的少量固定前缀（首帧 token），在局部窗口注意力下维持长程稳定。
抽象世界场景图
高精地图（车道线、路杆等）+ 动态物体 3D 立方体框组成的结构化条件，编码几何与未来轨迹。
FVD
Fréchet Video Distance，分布级视频质量指标，越低越好。
Cosmos / Cosmos-Predict 2.5
NVIDIA 视频基础模型平台，OmniDreams 的基座。
AlpaSim / Alpamayo / NuRec
分别是 NVIDIA 开源 AV 仿真编排器 / 开源 AV 策略栈 / 神经重建渲染器（被 OmniDreams 替换）。
SV / MV
单视角 / 多视角（4 路：前、左交叉、右交叉、前长焦；最多支持 7 路环视）。
